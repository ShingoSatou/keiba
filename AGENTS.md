# AGENTS.md

このファイルは、このリポジトリで あなた が従う永続的な指示を定義します。
作業を始める前に、以下のルールに従ってください。

---

## 0. 優先順位と適用範囲
- このディレクトリ配下の作業では、本ファイルの指示を適用する。
- ただし、より深い階層に `AGENTS.md` または `AGENTS.override.md` がある場合は、そちらを優先する。
- ユーザーの依頼内容に沿って作業し、明確な理由なく対象範囲を広げない。
- プロジェクト共通ルールと、各バージョン固有ルールの両方を尊重する。

---

## 1. プロジェクト方針（最重要）
### Keiba Prediction の前提
- 本リポジトリは **v1（旧）/ v2（安定）/ v3（最新）** が同居している。
- **新規開発は原則 v3** で行う。
- **v2 は互換維持・保守が中心** とする。
- **v1 は参照専用** とし、保守・編集しない。

### 作業前の確認ルール
- v3 の実装・変更を行う場合は、毎回以下を確認すること。
  - `docs/specs_v3/v3_システム仕様書.md`
  - `docs/specs_v3/` 内の対象作業に対応する詳細仕様書
  - `docs/ops_v3/`
- v3 の仕様書確認は、以下の台帳に従うこと。

| 作業内容 | 主に確認する仕様書 | あわせて確認する運用文書 |
|---|---|---|
| 全体構成、影響範囲判断、責務分割 | `docs/specs_v3/v3_システム仕様書.md`, `docs/specs_v3/v3_01_全体アーキテクチャ.md` | `docs/ops_v3/Assumptions.md` |
| 特徴量生成、odds snapshot、features table 列定義 | `docs/specs_v3/v3_02_特徴量生成とオッズ仕様.md` | `docs/ops_v3/Assumptions.md`, `docs/ops_v3/スクリプトリファレンス.md` |
| binary 学習、feature registry、Benter R*、odds 校正 | `docs/specs_v3/v3_03_二値分類と校正仕様.md` | `docs/ops_v3/Assumptions.md`, `docs/ops_v3/スクリプトリファレンス.md`, `docs/ops_v3/v3_run_report_*.md` |
| PL 学習、1レース推論、wide backtest | `docs/specs_v3/v3_04_PL推論とワイドバックテスト仕様.md` | `docs/ops_v3/Assumptions.md`, `docs/ops_v3/スクリプトリファレンス.md`, `docs/ops_v3/v3_run_report_*.md` |
| 共通 utility、CV、テスト、依存関係、CLI | `docs/specs_v3/v3_05_共通基盤と付録.md` | `docs/ops_v3/スクリプトリファレンス.md` |
- v2 を触る場合は、毎回以下を確認すること。
  - `docs/specs_v2/競馬予測システム 課題設定・要件定義書.md`
  - 必要箇所のみの確認でよい
- v3 は、v2 の作法を踏襲する前提で実装する。
  - Rolling 年次 CV
  - OOF 保存
  - as-of リーク防止
- そのため、v3 作業でも必要に応じて `docs/specs_v2/` を参照すること。

## 2. 対象ディレクトリの目安
### v1
- `docs/*_v1/`
- `scripts_v1/`
- `tests_v1/`
- `migrations_v1/`

### v2
- `docs/specs_v2/`
- `app/`
- `frontend/`
- `scripts_v2/`
- `test_v2/`
- `migrations_v2/`

### v3
- `docs/specs_v3/`
- `docs/ops_v3/`
- `scripts_v3/`
- `test_v3/`

### 判定ルール
- 新規機能追加、最新仕様対応、学習改善、評価改善は **まず v3 を検討** する。
- 既存 API/UI の軽微修正や互換維持は **v2 の可能性を検討** する。
- v1 は参照以外では触らない。

---

## 3. 作業スタイル
- 以下のような、軽微でない作業では、最初に短い計画を立てること。
  - おおむね 3 ステップ以上になる作業
  - 設計や構成に影響する変更
  - スキーマ / API の変更
  - 依存関係の変更
  - 大きめのリファクタリング
- 新しい情報によって計画が崩れた場合は、そのまま進めず、いったん立ち止まって計画を見直す。
- 進捗共有は短く具体的に行い、以下を中心に伝える。
  - 何を変更したか
  - 何を確認できたか
  - 何がまだリスクか
- 依頼内容が曖昧な場合は、すぐに質問するのではなく、周辺コード、テスト、ドキュメント、既存パターンを見て解釈を補う。
- 実装前に「どのバージョンを触るべきか」を必ず判断する。

---

## 4. コード変更の原則
- 常にシンプルさを優先し、問題を十分に解決できる最小の変更を選ぶ。
- 表面的な対処ではなく、根本原因を直す。
- ユーザーが明示的に回避策を求めていない限り、その場しのぎの修正はしない。
- 影響範囲を最小化する。
  - 無関係なリファクタリングをしない
  - 不要な rename をしない
  - 不要な整形差分を出さない
  - ついでの掃除をしない
- 新しい抽象化を入れる前に、まずリポジトリ内の既存パターンに従う。
- 明確な必要性がない限り、新しい依存関係は追加しない。
- v2 / v3 の流儀を混在させない。
- v3 では、仕様書・運用 docs に沿わない独自実装を避ける。

---

## 5. 安全ガード
- 以下のような影響の大きい変更は、実施前に確認を取ること。
  - 依存関係の追加・削除
  - 公開 API の変更
  - DB スキーマ変更やマイグレーション
  - CI/CD、インフラ、デプロイ設定の変更
  - 認証、シークレット管理、権限まわりの変更
  - 破壊的なファイル操作
  - v2 から v3 への責務移動
- 自分が作成していないユーザーの作業内容は、明示的な指示がない限り、上書き・削除しない。
- 危険性のあるコマンドが必要な場合は、実行前に理由を説明する。
- `data/`, `models/`, `.env` は Git 管理対象外である前提を守る。
- 機密情報、ローカル学習成果物、個人環境依存ファイルはコミットしない。

---

## 6. ファイルの読み方と調査順
- 最初は、必要最小限の関係ファイルだけを読む。
- 優先順位は以下の通り。
  1. 失敗しているテスト / エラーログ
  2. エントリーポイント
  3. 直接関係するモジュール
  4. 周辺ドキュメント / README / 既存実装パターン
  5. 対象バージョンの仕様書・運用 docs
- 必要がない限り、リポジトリ全体を無差別に読み回らない。
- v3 作業では、コードより先に `docs/specs_v3/` と `docs/ops_v3/` の整合を確認する。
- v2 作業では、既存 API/UI 互換を壊さないことを優先する。

---

## 7. 開発コマンド
### 共通
- 依存同期: `uv sync`

### v2
- API 起動: `uv run uvicorn app.main:app --reload`
- UI 起動: `cd frontend && npm ci && npm run dev`

### v3
- 追加依存（必要に応じて）: `uv sync --extra xgboost --extra catboost`
- 備考: PL 学習の torch は optional
- 特徴量生成: `uv run python scripts_v3/build_features_v3.py`
- 学習 / 評価の入口: `docs/ops_v3/スクリプトリファレンス.md`
- 実行結果レポート: `docs/ops_v3/v3_run_report_*.md`

### 実行ルール
- ad-hoc なコマンドより、まず既存スクリプトや docs で定義された入口を優先する。
- 学習・評価系は、勝手に独自の入口を増やす前に既存運用との整合を確認する。

---

## 8. v3 実装ルール
- v3 の変更時は、毎回 `docs/specs_v3/v3_システム仕様書.md` を入口にし、対象作業に対応する分割仕様書を基準とする。
- 学習・評価・特徴量生成・レポート出力は、`docs/ops_v3/` の運用に合わせる。
- 既存仕様との差異が出る場合は、コードだけ先に直さず、仕様差分を明示する。
- v2 由来の以下の原則を守る。
  - Rolling 年次 CV
  - OOF 保存
  - as-of リーク防止
- 特にリーク防止は最優先事項として扱う。
- 特徴量生成・学習・推論で、時点整合性を崩す変更は避ける。

---

## 9. v2 実装ルール
- v2 は保守・互換維持が中心であり、仕様逸脱を避ける。
- 作業前に `docs/specs_v2/競馬予測システム 課題設定・要件定義書.md` を参照する。
- API / UI / script の修正は、既存利用者への影響を最小にする。
- v3 の新設計を安易に v2 に逆輸入しない。

---

## 10. 完了前の確認
- 確認が終わるまで、作業完了とみなさない。
- まずは最小限で十分な確認を行い、必要に応じて確認範囲を広げる。
- 振る舞いを変えた場合は、可能なら関連するテスト、lint、型チェックを実行する。
- 学習・特徴量生成・評価に関わる変更では、可能なら以下を確認する。
  - 対象スクリプトが起動すること
  - 入出力パスが期待通りであること
  - リークや OOF 欠落を起こしていないこと
  - レポートや成果物の保存先が運用と一致していること
- 最終報告では、必ず以下を含める。
  - 何を変更したか
  - 何を確認したか
  - 実行したコマンド
  - 残っているリスクや未確認事項
- 確認を実行できなかった場合は、その理由を明確にし、手動確認の手順を示す。

## 11. バグ修正ポリシー
- バグ報告を受けたら、まず自分で調査し、そのまま修正まで進めることを基本とする。
- 調査は以下から始める。
  - ログ
  - スタックトレース
  - 失敗テスト
  - 再現手順
  - 対象バージョンの仕様書
- ユーザーの手間を増やさない。リポジトリ内で自分が確認できることは、できるだけ自分で調べる。
- CI 失敗が見えていて、かつ今回の修正範囲に含まれるなら、合わせて対処する。
- v3 の不具合では、コードだけでなく `docs/specs_v3/` / `docs/ops_v3/` との不整合も疑う。

---

## 12. 品質基準
- 解決案を提示する前に、次を自問する。
  - これは最もシンプルで正しい修正か？
  - シニアエンジニアが見ても納得できる差分か？
  - 必要な箇所だけを変更しているか？
  - 既存の慣習や流儀を守っているか？
  - v2 / v3 の境界を壊していないか？
- 重要な変更では、一度立ち止まって、もっときれいな設計がないか検討する。
- 単純な問題に対して、過剰設計をしない。

## 13. タスク管理
- 複数ステップの作業では、`tasks/todo.md` を台帳として使う。
- `tasks/todo.md` には task の要約とリンクだけを書き、1画面で見渡せる分量を維持する。
- 実際の作業内容は 1 task = 1 file で管理し、新規 task は `tasks/templates/task.md` を元に `tasks/active/` へ作成する。
- task file には最低限、以下を記録する。
  - 対象範囲
  - 対象バージョン（v2 / v3）
  - 前提・仮定
  - チェックリスト
  - 確認結果
  - 実行コマンド
  - 残リスク
- task 完了後は `tasks/done/YYYY/` へ移動し、`tasks/todo.md` の台帳リンクを更新する。
- 同じテーマでも完了基準が異なるなら別 task file に分ける。
- ユーザーからの修正指摘や、同じミスの再発があった場合は、`tasks/lessons.md` に短い再発防止ルールを追記する。
- 関連作業に着手する前に、必要に応じて `tasks/todo.md` と `tasks/lessons.md` を見直す。

## 14. マルチエージェント利用
- マルチエージェントが有効で、かつ作業を並列化できる場合は、独立した小タスクごとにエージェントを分ける。
- 例：
  - コードベース調査
  - 実装案の比較
  - テスト失敗の切り分け
  - ドキュメント確認
  - v2 / v3 の仕様差分確認
- 1 エージェントにつき 1 サブタスクを原則とする。
- メインの流れでは、統合と意思決定に集中する。

---

## 15. 言語別・実装別ガイダンス
- 既存の命名、モジュール構成、公開インターフェースは、依頼上必要でない限り変えない。
- コメントは、本当に理解の助けになる箇所にだけ書く。自明なコードの説明はしない。
- 振る舞いを変えた場合は、テストも更新する。
- 公開仕様、セットアップ方法、利用方法が変わった場合は、ドキュメントも更新する。
- データ処理・特徴量生成・学習ロジックでは、再現性と時系列整合性を重視する。

---

## 16. 最終報告の形式
- 最終報告は簡潔かつ構造的にまとめる。
  1. 変更概要
  2. 変更ファイル
  3. 実施した確認
  4. 残リスク / 次の対応
