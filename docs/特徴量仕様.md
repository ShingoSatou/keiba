# 特徴量仕様書（単勝） v1.2

## 1. 目的と出力

* **目的**：各レースの各馬について **単勝=1着確率** (p(win)) を推定する
* **学習ラベル**：成績確定後の「1着=1 / それ以外=0」
* **推論タイミング**：**発走時刻の5分前（T-5）** を基準

  * ※オッズは時系列で入るため、**T-5以前で最も近い時点のスナップショット**を使う
  * 特徴量の求め方は、 **特徴量カタログ.md**を参照

---

## 2. モデル構成（2本立て＋Ensemble）

あなたの理解で合っています。運用上も堅いです。

> 呼称メモ：本書の **Model A = Model F（Fundamental）**、**Model B = Model M（Market）**。他の仕様書では F/M 表記を使うことがあります。

### Model A：オッズ・票数なし（Fundamental）

* 市場情報（オッズ/票数）を入れない
* **競走能力・適性・状態・相手関係**で勝ち切れる馬を推定

### Model B：オッズ・票数あり（Market）

* **オッズ（＋票数）を特徴量に含める**
* 市場の集合知を取り込みつつ、歪み（過小評価）を捉える

### Ensemble（推奨：stacking）

* 入力：Model Aの確率、Model Bの確率（＋補助特徴量少数）
* 出力：最終確率
* 期待値最大化での安定性が上がりやすい（片方が崩れた日の耐性）

## 3. 取得レコード（JV-Link/JV-Data）—最低限セット

### 重要：Backfillable / Realtime-only（学習期間との整合）

* **Backfillable（再現学習に使える）**：2016〜現在を再現可能に学習・検証できる
  * JVOpen（蓄積系）が中心
  * **0B41（時系列オッズ）は JVRTOpen の racekey 指定でバックフィル**する（参照: `docs/ダウンロードリスト.md`, `docs/jvlink_handover.md`）
* **Realtime-only（運用開始以降のみ）**：運用当日の監視/スナップ用途で取得するが、長期バックフィルは前提にしない
  * 例：0B11（WH）, 0B14（WE/AV/JC/TC/CC）, 0B13/0B17（速報DM/TM）

> つまり「学習期間=2016〜現在」は、Model F（Fundamental）はJVOpen最小セットで成立し、
> Model M（Market）/ Model O（最終オッズ予測）は **0B41バックフィル**により同様に 2016〜現在で学習・検証できる。
> 一方で 0B11/0B14/0B13/0B17 を学習・検証で使う場合は、採用範囲/バックフィル方針を別途仕様化する。

### 3.1 速報/リアルタイム系（当日T-5推論で必要）

* **0B41（時系列オッズ：単複枠）**＝中間オッズを時系列で収録（レコードは `O1`） ([EveryDB2 - JRA-VANデータベース構築の決定版！][1])
* **0B11（速報馬体重）**（レコードは `WH`）
* **0B14（当日変更）**（`WE/AV/JC/TC/CC`）
* **0B13/0B17（速報DM/TM）**：直前予想は「馬体重発表後」、最新予想発表ごとに上書き更新

> 0B41/0B42の“間隔”は仕様書に明記されないことがありますが、開発者コミュニティの過去ログでは実データ上「ほぼ5分間隔」との観測が語られています（検証済み前提で扱うのが安全）。 ([forum.jra-van.ne.jp][2])

### 3.2 蓄積系（過去成績・調教・各種統計）

* **RACE / DIFF / MING / SNPN（旧SNAP→新SNPN） / SLOP / WOOD / COMM / YSCH** など（後述の統計・調教・出走時点情報に使用）
* 出走時点情報は **“SNAP → SNPN” に変更**（Ver4.9.0以降の仕様変更に注意） ([JRA-VAN Data Lab.開発者コミュニティ][3])
* 血統（BLOD/BLDN 等）は現状の最小構成では未使用（必要になったら追加）

## 4. 時系列オッズ（O1 / 0B41）の厳密ルール（重要）

### 4.1 O1のデータ区分（どれを使うか）

O1（単複枠）はデータ区分で **中間/最終/確定**などが分かれます：
**1:中間 / 2:前日売最終 / 3:最終 / 4:確定 / 5:確定(月曜)**

### 4.2 発表月日時分（時系列キー）

O1には **「発表月日時分」**があり、これは

* **中間オッズのみ設定**
* **時系列オッズを使用する場合のみキーとして設定**
  と明記されています

→ したがって **T-5スナップショット**は、

* **データ区分=1（中間）**のレコードから
* `発表月日時分 <= 発走時刻-5分` の中で **最大（最も近い）** を採用
  が仕様に沿った定義です。

### 4.3 O1オッズの特殊値と欠損扱い（重要）

O1（単勝オッズ: 4桁）は、実データ上 `****` 等の記号が混入し得るため、**欠損扱いを明示**する。

* 特殊値（単勝）:
  * `"9999"`：999.9倍以上
  * `"0000"`：無投票
  * `"----"`：発売前取消
  * `"****"`：発売後取消
  * `" "`：登録なし
* 取込（`core.o1_win.win_odds_x10`）:
  * `*` を含む値、数字が無い値は `NULL`（欠損）として保存
  * `"0000"` は `0` として保存され得る
* T-5側（スナップ/推論）:
  * `win_odds_x10 IS NULL OR win_odds_x10 <= 0` は欠損扱い
  * `odds_win_t5=NULL` / `odds_missing_flag=true`
  * 推論は欠損オッズ行を `skip` にする（`odds_win_t5<=0` も欠損扱い）

# 5. 特徴量一覧（確定）

以下、**「導入する」前提で確定**します（消す方が楽方針）。

---

## 5A. 共通キー・基礎（Model A/B共通）

| Feature群 | 特徴量                         | 定義                  |
| -------- | --------------------------- | ------------------- |
| ID       | race_id / horse_id / umaban | 開催年+月日+場+R + 馬番     |
| 学習用      | y_win                       | 1着=1 else 0（推論時は無し） |
| 分割       | race_date                   | 時系列CV用（リーク防止）       |

## 5B. レース条件（Model A/B共通）

**出典**：RA（レース詳細）、WE（天候馬場）など

導入する特徴量（例：すべてone-hot/カテゴリ埋め込み可能）

* 距離、芝/ダ、右左/直線、コース種別、クラス/重賞格
* 天候コード、馬場状態（良/稍/重/不良など）
* 出走頭数（登録頭数・出走頭数）
* 発走時刻（変更があればTC反映）
* 取消・除外（AV）/騎手変更（JC）/コース変更（CC）フラグ

> 速報系一覧にWE/AV/JC/TC/CCが含まれることは仕様書の速報系データ種別一覧にも整理されています。 ([EveryDB2 - JRA-VANデータベース構築の決定版！][1])

---

## 5C. 出走馬の当日情報（Model A/B共通）

**出典**：SE（出馬表）、WH（馬体重） ([EveryDB2 - JRA-VANデータベース構築の決定版！][1])

* 枠番、馬番
* 斤量
* 騎手コード、調教師コード
* 馬体重、増減（**WHの最新<=T-5 を優先**。無ければ **SE内の `body_weight/body_weight_diff`** をフォールバック）
* ブリンカー等の装備フラグ（取れるなら）

---

## 5D. 近走成績・パフォーマンス集約（Model A/B共通）

**方針**：直近N走（推奨：5走＋12ヶ月）を基本に、距離/馬場/競馬場の類似条件で分けて集約。

導入（すべて数値特徴量）：

* 直近N走：着順、走破タイム、上がり、通過順、人気、馬体重
* **間隔（休養）**：前走からの日数、長期休養フラグ
* **適性分解**：芝/ダ、距離帯（短/マ/中/長）、右左、坂、季節
* **展開寄り**：前半位置取り指標（逃/先/差/追の傾向）
* **安定性**：パフォーマンス分散、凡走/好走の振れ幅

---

## 5E. スピードレーティング（Model A/B共通・強推奨）

### 導入する値（確定）

* `sr_last`（前走SR）
* `sr_best_12m`（12ヶ月最高SR）
* `sr_mean_5`（直近5走平均SR）
* `sr_trend_3`（直近3走の傾き）
* `sr_course_dist`（同コース×距離のSR平均）

> ここでは「JRA-VANに既にレーティングがあるならそれを使うべきか？」という点は、**“あるなら併用”**が最適解です。
>
> * **既存値をそのまま特徴量として入れる（楽で強い）**
> * さらに **自作SRも入れる（定義が違うため情報が残る）**
>   で「消す方が楽」方針に合います。

---

## 5F. 出走別着度数 CK（Model A/B共通・導入）

**役割**：騎手・調教師・馬などの「条件別の着度数」＝強い事前統計。
また、取得のデータ種別は **SNAPではなくSNPN**に変更された点が重要です。 ([JRA-VAN Data Lab.開発者コミュニティ][3])

### 導入方針（確定）

* CKの提供項目は **原則すべて取り込み**
* 特徴量化は以下を作る（すべて導入）

  * **生の度数**（1/2/3着/着外など）
  * **率**（勝率/連対率/複勝率）
  * **標本サイズ**（母数、過去出走数）
  * **ベイズ平滑化率**（母数が小さい時の暴れ対策）

---

## 5G. 調教（SLOP/WOOD等：Model A/B共通・導入）

導入（確定）：

* 最終追い切りの時計、ラップ形状指標（加速/減速）
* 追い切り本数（過去X週）
* コース種別（坂路/ウッド等）
* 併せ馬/単走、評価が取れるなら評価も

運用メモ（2026-02-20 時点）:
* 2016+ 学習データでは WOOD 欠損率が高いため、ベースライン学習の入力から `wood_*` は一時除外する。
* 条件近似の `*_sim_*` / `n_sim_runs_5` も欠損率が高いため、ベースライン学習の入力から一時除外する。
* SLOP は継続利用し、WOOD はデータ期間を絞った比較実験で再評価する。

## 5H. JRA-VANデータマイニング予想（DM/TM）（Model A/B共通・導入）

**速報DM（0B13）**は「直前予想（馬体重発表後）」で、最新予想発表ごとに上書き更新されます
**TM**は「予測スコア（0〜100）」等が入ることが仕様書にあります

### 導入（確定）

* **DM**：予想走破タイム、予想誤差（±）、予想順位
* **TM**：予測スコア、予測順位
* **A/B両方に入れる**（市場オッズとは別系列の外部推定）

# 6. Market特徴量（Model Bのみ：オッズ・票数あり版）

> 注意：本章（Market特徴量）は **0B41（時系列オッズ）に依存**します。運用で使うなら **学習・検証用にも 0B41 をバックフィル**して整合させる（参照: `docs/ダウンロードリスト.md`）。

## 6A. T-5オッズスナップショット（必須）

**出典**：0B41（時系列オッズ：単複枠） ([EveryDB2 - JRA-VANデータベース構築の決定版！][1])

導入（確定）：

* `odds_win_t5`：単勝オッズ（T-5の最新）
* `imp_prob_t5 = 1/odds_win_t5`
* `log_odds_t5 = log(odds_win_t5)`
* `rank_odds_t5`：オッズ順位
* `entropy_t5`：レース全体の確率分布エントロピー（荒れ度）

## 6B. オッズの動き（時系列特徴量：必須）

0B41は中間オッズが時系列で入る前提なので、**直近K点**（例：T-60〜T-5の最大12点）で以下を導入：

* `odds_slope`：回帰直線の傾き（下がる=買われる）
* `odds_volatility`：標準偏差 / 変動係数
* `odds_min/max`：期間内の最小/最大
* `odds_last_change`：直近1ステップの変化率
* `odds_jump_flag`：急変（閾値超え）フラグ

## 6C. 票数（入る範囲で導入）

O1には票数合計項目が存在します（例：単勝票数合計など）。

導入（確定）：

* `total_bets_win_t`：その時点の単勝票数合計（取得できる範囲）
* `bet_share_t5`：票数シェア（その馬の票数/総票数）※馬別票数が取れるなら
* `odds_vs_bets_inconsistency`：オッズと票数のねじれ指標（人気の歪み検出）

# 7. 「最終オッズ予測モデル」を別に作って入れるべきか？

結論：**効果が出る可能性は高い**ので、あなたの案（別モデリング→特徴量/EVへ投入）は筋が良いです。理由はシンプルで、

* 単勝の期待値は最終オッズで決まる
* T-5のオッズは最終の近似だが、**締切間際の資金流入**で動く
  → その“残りの動き”を当てられると、EV計算のブレが減る

実際、時系列オッズ（0B41/0B42）を使う話題や、取得間隔（ほぼ5分）に関する議論は開発者コミュニティでも行われています。 ([forum.jra-van.ne.jp][2])

### ただし注意（設計に書いておくべき現実）

* 0B41の提供は「時系列オッズ（単複枠）」として定義されていますが ([EveryDB2 - JRA-VANデータベース構築の決定版！][1])
  **長期バックテスト（2016〜）で必要な粒度の時系列を揃えられるか**は、あなたのDBにどこまで蓄積済みか（or 取得可能データの実体）に依存します。
  → ここは「実データで確認して進める」というあなたの方針が正解です。

# 8. 実装のための「特徴量生成ルール」—最低限の取り決め

## 8.1 as-ofルール（リーク防止）

* すべて **発走時刻-5分以前**の最新情報に丸める
* TC（発走時刻変更）があれば **発走時刻そのものを更新**してT-5を再計算
* DM/TMは「馬体重発表後の直前予想」で更新され得るので
  **T-5以前の最新**を採用

## 8.2 欠損処理（確定）

* 数値欠損：`missing_flag`を立て、値は中央値（または0）埋め
* 度数系（CK等）：母数0なら率は欠損→0、別途`n=0`フラグ
* オッズ欠損：発売前/取消などは仕様上の記号が入り得るため、数値化できない場合は欠損扱い
  * 具体ルールは「4.3 O1オッズの特殊値と欠損扱い」を参照
