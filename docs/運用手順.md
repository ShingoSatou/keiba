# 運用手順（非開発者向け）

この文書は「コマンドを実行して動かす」ための手順書です。仕様の背景は `docs/運用仕様書.md` を参照してください。

## 0. 前提（最重要）

- DB: PostgreSQL が必要（Raw/Core/Mart を使用）
- 取得（JV-Link）: **Windows + JV-Link + 32bit Python** が必要
- 実行（ETL/特徴量/推論）: WSL/Linux の **64bit Python（uv）** で実行

## 1. セットアップ（最初に1回）

### 1.1 Python（uv）

```bash
uv sync
# W&B を使う場合のみ
uv sync --extra wandb
```

### 1.2 DB作成 + migrations

```bash
# DB作成（例: keiba_cp）
bash setup_postgres_multi.sh keiba_cp

# スキーマ作成/更新（migrations適用）
uv run python scripts/migrate.py
```

`.env` に `DATABASE_URL` を設定します（例: リポジトリ同梱の `.env` を編集）。

### 1.3 data ディレクトリ（置き場所）

このリポジトリは「リポジトリの1つ上」に `data/` を置く運用を前提にしています（例: `../data`）。

```bash
mkdir -p ../data
ln -snf ../data data
```

※ `--output-dir` / `--input-dir` / `--rt-data-dir` で別場所も指定できます。

### 1.4 Windows 32bit Python（JV-Link用）

WSL から当日運用（`ops_*`）を回す場合、Windows 側の 32bit Python を呼び出します。

- `PYTHON32_PATH`（任意）: Windows 32bit Python の `python.exe` を指定  
  - 例: `C:\\path\\to\\REPO-cp\\.venv32\\Scripts\\python.exe`

`\.venv32` が無い場合の例（PowerShell）:

```powershell
# 32bit Python が入っている前提（例: py -3.11-32）
py -3.11-32 -m venv .venv32
.\.venv32\Scripts\python.exe -m pip install -U pip
.\.venv32\Scripts\python.exe -m pip install pywin32
```

#### 蓄積データ取得（Windows / 32bit Python）

JVOpen（蓄積系）の取得は Windows 32bit Python で実行します（出力は `data/*.jsonl`）。

```powershell
# マスタ（初回に推奨）
.\.venv32\Scripts\python.exe scripts/extract_jvlink.py --dataspec DIFF --from-date 20160101 --option 4

# レース/成績（例: 2年刻み）
.\.venv32\Scripts\python.exe scripts/extract_jvlink.py --dataspec RACE --from-date 20160101 --to-date 20171231 --option 4

# 学習DM/TM（蓄積）
.\.venv32\Scripts\python.exe scripts/extract_jvlink.py --dataspec MING --from-date 20160101 --option 4

# CK（出走別着度数）
.\.venv32\Scripts\python.exe scripts/extract_jvlink.py --dataspec SNPN --from-date 20160101 --option 4
```

推奨dataspec（SLOP/WOOD/COMM/YSCH など）や運用上の注意は `docs/ダウンロードリスト.md` / `docs/データ取得仕様書.md` を参照してください。

## 2. data/ の「SQLに取り込むもの / 取り込まないもの」

### 2.1 SQLに取り込む（`scripts/load_to_db.py`）

| 種別 | ファイル例（prefix） | 取り込み先（主） | 用途メモ |
|---|---|---|---|
| 蓄積（マスタ差分） | `DIFF_*.jsonl` | `raw.jv_raw`, `core.*(horse/jockey/trainer...)` | 初回に推奨 |
| 蓄積（レース/成績） | `RACE_*.jsonl` | `raw.jv_raw`, `core.race/core.runner/core.payout/core.odds_final` | 学習の母集団 |
| 蓄積（学習DM/TM） | `MING_*.jsonl` | `raw.jv_raw`, `core.mining_dm/core.mining_tm` | 学習は基本これ |
| 蓄積（CK） | `SNPN_*.jsonl` | `raw.jv_ck_event`, `core.ck_runner_event`, `mart.feat_ck_win` | `raw.jv_raw` は使わない |
| 蓄積（調教） | `SLOP_*.jsonl`, `WOOD_*.jsonl` | `raw.jv_raw`, `core.training_slop/core.training_wood` | 任意（欠損許容） |
| RT（時系列オッズ） | `0B41_*.jsonl` | `raw.jv_raw`, `core.o1_header/core.o1_win` | Model M/O で必須 |
| RT（馬体重） | `0B11_*.jsonl` | `raw.jv_raw`, `core.wh_header/core.wh_detail` | 当日T-5 |
| RT（変更） | `0B14_*.jsonl` | `raw.jv_raw`, `core.event_change` | 当日T-5（重要） |
| RT（速報DM/TM） | `0B13_*.jsonl`, `0B17_*.jsonl` | `raw.jv_raw`, `core.rt_mining_dm/core.rt_mining_tm` | 当日T-5（欠損は許容運用可） |

ロード:

```bash
# まとめて投入（ディレクトリ指定）
uv run python scripts/load_to_db.py --input-dir data/

# 1ファイルだけ投入
uv run python scripts/load_to_db.py --input data/RACE_20260207_123456.jsonl

# 0B41 だけまとめて投入（ファイル数が多い場合は wildcard 推奨）
uv run python scripts/load_to_db.py --input "../data/0B41_*.jsonl"

# 0B41 を「複数ファイル → 1ファイル」に結合済みなら、その1ファイルだけ投入できる。ファイル数が多く、ロードに時間を要する場合は推奨。
# 例: ../data/0B41_ALL_20260215.jsonl（prefix が 0B41_ で始まる必要あり）
uv run python scripts/load_to_db.py --input ../data/0B41_ALL_20260215.jsonl
```

### 2.2 いまはSQLに取り込まない（または raw のみで未利用）

| 種別 | ファイル例（prefix） | 状態 |
|---|---|---|
| コース情報 | `COMM_*.jsonl` | 取得は可能だが **後工程（core化/特徴量利用）が未整備**（ロードしても `raw.jv_raw` に入るだけ） |
| 開催スケジュール | `YSCH_*.jsonl` | 同上 |
| 学習用ファイル | `train*.parquet`, `*dataset*.parquet` | DBから作る成果物。SQLへは戻さない |
| 運用出力 | `data/ops/<YYYYMMDD>/*` | CSV/JSON/HTML/監査ログ。SQLへは戻さない |

## 3. 当日運用（最短：T-5推論まで）

前提:
- DBに当日 `core.race/core.runner` が揃うこと（不足時は自動補完可）
- `models/t5_bundle.pkl` が存在すること（なければ「5. モデル作成」を先に実行）

```bash
# 例: 2026/02/14 の当日運用（取得→投入→スナップ→推論→出力）
uv run python scripts/ops_t5_dryrun.py \
  --race-date 20260214 \
  --feature-set realtime \
  --fetch-rt \
  --fetch-rt-policy auto \
  --fail-on-empty
```

出力（デフォルト）:
- `data/ops/<YYYYMMDD>/t5_snapshot.{csv,json,html}`
- `data/ops/<YYYYMMDD>/t5_predictions.{csv,json}`
- `data/ops/<YYYYMMDD>/t5_audit.json`
- `data/ops/<YYYYMMDD>/run_meta.json`

## 4. 当日運用（手動で分けて実行したい場合）

```bash
# 1) 当日RACE（不足時のみでOK）
uv run python scripts/ops_race.py --race-date 20260214

# 2) 速報取得（WSL→Windows32bit）
uv run python scripts/ops_rt.py --race-date 20260214

# 3) 速報JSONL→DB投入
uv run python scripts/ops_rt_load.py --race-date 20260214

# 4) T-5 スナップ生成（必要なら --ensure-race をONのまま）
uv run python scripts/build_t5_snapshot.py --date 2026-02-14 --feature-set realtime

# 5) 推論
uv run python scripts/predict_t5.py --race-date 2026-02-14 --feature-set realtime
```

※ `ops_*` は `--race-date` が `YYYYMMDD`、`build_t5_snapshot/predict_t5` は `YYYY-MM-DD` です。

## 4.5 学習データ再生成と品質ゲート

学習・バックテストを実行する前に、`data/train.parquet` を再生成して品質ゲートを通します。

```bash
# 1) core.race の補正（stub/距離異常の再投入）
uv run python scripts/backfill_core_race.py --from-date 2016-01-01 --only-bad

# 2) mart を再構築（学習対象期間）
# デフォルト: 障害レース(surface=3)は除外（含める場合は --include-obstacles）
uv run python scripts/build_features.py --rebuild --from-date 2016-01-01

# 3) train.parquet を再生成
# デフォルト: 障害レース(surface=3)は除外（含める場合は --include-obstacles）
uv run python scripts/build_dataset.py --from-date 2016-01-01 --min-horses 5 --output data/train.parquet

# 4) 品質ゲート（fail-fast）
uv run python scripts/check_dataset_quality.py --input data/train.parquet --gate --json-output data/quality/train_latest.json
```

`QUALITY_GATE: FAIL` の場合は、`data/quality/train_latest.json` と `docs/データ品質レポート.md` を確認してから再実行します。

## 5. モデル作成（t5_bundle.pkl）

walk-forward で学習・検証し、運用バンドル（`models/t5_bundle.pkl`）を作ります。

```bash
# 例: walk-forward 実行（出力先は scripts 側のデフォルトに従う）
uv run python scripts/train_walk_forward.py

# 例: 運用バンドル作成
uv run python scripts/train_production_bundle.py
```

プロジェクトの現状/制約（欠損や当日データ品質など）は `docs/進捗状況.md` を参照してください。
