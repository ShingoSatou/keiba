# 運用仕様書（単勝・T-5）v1.0

## 1. 目的

* 当日（開催日）に **T-5時点の情報**で単勝の勝率・期待値を算出し、**買い判断**を出力する（手動購入前提）。
* 長期（2016〜現在）の学習データを更新し、モデルを定期的に再学習・リリースする。
  * ただし Model M / Model O（0B41依存）は、**0B41バックフィル**により 2016〜現在で学習・検証・更新できる（参照: `docs/ダウンロードリスト.md`, `docs/jvlink_handover.md`）。
  * 0B11/0B14/0B13/0B17 を学習・検証で使う場合は、採用範囲/バックフィル方針を別途仕様化する。
* 運用の再現性（「その日なぜその馬を買ったか」）を監査ログとして残す。

---

## 2. 運用範囲（In/Out）

### In

* 当日推論：O1時系列、馬体重、変更、速報DM/TM を用いた **T-5推論**
* 買い判断：EV閾値 τ に基づく購入候補出力
* 学習更新：蓄積データの取得、特徴量マート更新、walk-forward検証、モデル更新

### Out

* 自動投票（APIでの発券）
* WIN5最適化
* 外部ニュースやSNS等のリアルタイム取り込み

## 5. 当日運用（推論〜買い判断）仕様

## 5.1 当日フロー（全体）

**開催日ごとに、全レースを「発走時刻T」に対してT-5で処理**する。

1. **当日レースカレンダー生成**（発走時刻・出走馬）
2. **変更情報を逐次反映**（発走時刻変更、取消、騎手変更など）
3. 各レースについて **asof_ts = T-5** を計算
4. **asof_ts以前の最新スナップ**で特徴量作成
5. 予測：

   * Model F（オッズなし）→ `pF`
   * Model M（オッズあり）→ `pM`
   * Model O（最終オッズ予測）→ `ô_close`
   * stacking → `p_final`（校正器なし）
6. **EV計算** → 購入候補抽出（τ適用）
7. 出力（買い目表）＋監査ログ保存
8. オペレータが手動購入

---

## 5.2 取得仕様（当日：JVRTOpen）

当日推論に必要な速報取得は以下。0B41 は **レース単位**、それ以外は **開催日単位**での取得を基本とする（取得回数を抑える）。

* 0B41：時系列オッズ（O1：単勝を使用）
* 0B11：馬体重（WH）
* 0B14：変更（速報開催情報(一括): WE/AV/JC/TC/CC）
* 0B13：速報DM
* 0B17：速報TM

> 注: 0B16（速報開催情報(指定)）は **JVWatchEvent のイベント key** 前提のため、本リポジトリのポーリング運用では通常利用しない（必要なら別途イベント常駐の設計が必要）。

### 5.2.1 取得タイミング（推奨）

* **レース開始60分前〜**：0B41（オッズ系列を蓄積）
* **レース開始30分前〜**：0B11（馬体重）・0B14（変更）を監視
* **レース開始10分前〜**：0B13/0B17（DM/TM）を監視
* **T-5ちょうど**：最終スナップ確定→推論実行→出力

> 実装は「常駐でポーリング」か「レースごとジョブを時刻起動」どちらでもよい。
> ただし最終的に **T-5時点のスナップ採用キー**が残る形にする。

## 5.3 as-of（T-5）スナップ採用仕様（当日）

### 5.3.1 発走時刻（T）の確定

* レースの基準発走時刻 T は、当日変更（TC）があれば反映する。
* **TCが発生した場合はasof_tsを再計算**し、レース処理のスケジュールを更新する。

### 5.3.2 O1（時系列オッズ）採用

* 対象：中間（時系列）オッズ
* ルール：`announce_ts <= asof_ts` を満たす中で **announce_ts最大**のO1を採用
* 採用結果（監査ログに保存）：

  * `odds_snapshot_announce_ts`
  * `odds_snapshot_age_sec = asof_ts - announce_ts`

**購入抑制（安全弁）**

* `odds_snapshot_age_sec > 15分`（例）なら「オッズが古い」扱いで

  * そのレースは **出力に警告**
  * ルールにより **購入対象外**にしてもよい（設定可能）

### 5.3.3 DM/TM（速報）採用

* 優先順位：`kbn=3 → 2 → 1`
* 同一kbnが複数：**作成時刻が最大**を採用
* `kbn=0（削除）` は採用禁止
* 採用結果（監査ログに保存）：

  * `dm_kbn, dm_create_time`
  * `tm_kbn, tm_create_time`

### 5.3.4 馬体重（WH）採用

* `published_ts <= asof_ts` の最新
* 取得できない場合：欠損フラグを立てる（馬体重を理由に除外はしない）

### 5.3.5 取消/除外（AV）・騎手変更（JC）

* AVで「走らない」馬は **推論対象外**（出力対象外）
* JCは当日特徴量（騎手ID）を置換し、騎手変更フラグを立てる

## 5.4 推論仕様（当日）

### 5.4.1 モデル入出力

* 入力：Mart（T-5確定行：1行=1頭）
* 出力（各馬）：

  * `pF, pM, ô_close, p_final`
  * `EV_hat = p_final * ô_close - 1`
  * `odds_t5`（T-5オッズ）、`o_final`（当日は不明、後日検証用）

### 5.4.2 stacking（校正器なし）

* `p_final = meta_model(pF, pM, ô_close, …)`
* 出力はそのまま確率として扱う（校正器なし）

---

## 5.5 買い判断（意思決定）仕様

### 5.5.1 標準購入ルール（R1固定）

* 1レースで最大1頭
* 候補：当該レースで `EV_hat` 最大の馬
* 購入条件：`EV_hat >= τ`
* 賭け金：固定 `stake=100円`

### 5.5.2 除外・抑制条件（安全弁）

以下のいずれかに該当する場合、購入候補から除外（または警告付きで除外）：

* 取消/除外（AV）
* オッズスナップが古すぎる（例：15分以上更新なし）
* レース情報が不整合（出走頭数とO1が一致しない等）
* 当日モデル入力の欠損率が閾値超過（例：主要特徴量の欠損>30%）

## 5.6 出力仕様（オペレータが買える形）

**必ず2種類出す：人間用＋機械用**

### 5.6.1 人間用（推奨：HTML/PDF or 表形式）

レースごとに並べる列（例）：

* レースID、発走時刻、馬番、馬名
* `odds_t5`, `ô_close`, `p_final`
* `EV_hat`
* `候補フラグ（買う/見送り）`
* 注意フラグ：オッズ古い、DM/TM欠損、馬体重欠損、騎手変更 等

### 5.6.2 機械用（CSV/JSON）

* 1行=1頭、キー＋全推論結果＋採用スナップキーを含む
* 後日の検証ができるように最低限これを含める：

  * `race_key, horse_no, asof_ts`
  * `odds_snapshot_announce_ts, dm_kbn, tm_kbn, wh_ts`
  * `pF, pM, ô_close, p_final, EV_hat`
  * `stake, buy_flag, tau_version, model_version, feature_version`

## 6. 定常運用（データ更新・学習・リリース）

## 6.1 蓄積データ更新（JVOpen）

目的：学習DBの更新、後日検証のための確定値の取り込み

* 週次（または毎開催後）：

  * RACE（RA/SE/HR/H1/O1）
  * MING（学習用DM/TM）
  * SNPN（CK）
  * 調教（SLOP/WOOD：ある期間のみでもOK）

### 6.1.1 取り込みの原則

* Rawへ必ず保存（再パース可能にする）
* Coreへ正規化（キー衝突はUPSERT）
* Martを再生成（2016〜最新まで増分更新）

---

## 6.2 再学習サイクル

### 6.2.1 頻度（推奨）

* **月次**（最小）
* 可能なら **隔週**（概念ドリフト対策）

### 6.2.2 学習・検証の方法（検証設計仕様 v1.2 に従う）

* walk-forward outer fold
* inner split：Train-fit / ES-valid / Calib
* early stopping：ES-validで実施
* 校正器なし（ただしECE等は監視）

### 6.2.3 リリース判定（ゲート）

新モデルを本番に上げる条件（例）：

* OuterValidで **Logloss改善**（または同等）
* OuterValidで **ROI改善**（τ固定ルールで比較）
* 最大DDが悪化しない（許容範囲）
* 主要なデータ欠損率・処理時間が許容内

## 7. 監視（モニタリング）仕様

## 7.1 当日（リアルタイム）監視

	  * 取得監視：

	  * 0B41/O1が一定時間更新されない
	  * 0B14の変更が反映されない（発走時刻ズレ）
* 推論監視：

  * 推論時間がT-5に間に合わない
  * 欠損率が急増（入力異常）
* 出力監視：

  * buy_flagが極端（全レース買う/全レース買わない）

## 7.2 長期（モデル）監視

* データドリフト：

  * odds分布、距離分布、頭数分布の変化
* モデル劣化：

  * 月次でAUC/Logloss/Brier/ECE、ROI、購入回数
* しきい値 τ の安定性：

  * τ固定でROIが崩れる場合は再最適化サイクルを検討（ただし過学習注意）

## 8. 障害対応（Runbook）

### 8.1 取得失敗

* 0B41取得不能：そのレースは Model M と Model O を無効化し、**Model Fのみ**で推論（買いは原則停止でもよい）
* 0B14取得不能：発走時刻変更が反映できない → **レース処理を停止**（T-5がズレるため）
* 0B13/0B17欠損：DM/TMを欠損で実行（買い停止にはしない）
* 0B11欠損：馬体重は欠損許容で継続（監査/警告を残す）

### 8.2 DB遅延／ETL遅延

* T-5までにMartが作れない場合：

  * そのレースは「見送り」
  * 原因（クエリ遅延、ロック、I/O）を記録し、後日改善

### 8.3 自動判定（day-health）

`scripts/ops_t5_dryrun.py` は day-health を集計し、`run_meta.json` に `health` として保存する。

* 判定指標:
  * `missing_start_time`, `race_stub`, `o1_rows`, `wh_rows`, `event_rows`, `rt_dm_rows`, `rt_tm_rows`
  * `snapshot_odds_missing_rows`, `snapshot_odds_missing_races`（after_snapshot で評価）
* 停止（`status=stop`）:
  * `core.race=0` または `core.runner=0`
  * `missing_start_time / races >= 0.50`
  * `race_stub / races >= 0.50`
  * `0B14` が required なのに取得ファイルなし（`--fetch-rt-policy auto` 含む）
  * `--fail-on-empty` 指定時に `t5_snapshot rows=0`
* 警告（`status=warn`）:
  * 上記 stop 未満の欠損（例: `missing_start_time>0`, `race_stub>0`）
  * `0B41/0B11` 欠損（warn）
  * `0B13/0B17` 欠損（continue）
  * `snapshot_odds_missing_rows / t5_snapshot_rows >= 0.50`
  * `snapshot_odds_missing_races / races >= 0.50`

### 8.4 日次運用コマンド（推奨順）

1. 当日RACE補完（必要時）  
   `uv run python scripts/ops_race.py --race-date YYYYMMDD`
2. RT取得  
   `uv run python scripts/ops_rt.py --race-date YYYYMMDD --dataspecs 0B41,0B14,0B11,0B13,0B17`
3. RT投入  
   `uv run python scripts/ops_rt_load.py --race-date YYYYMMDD --policy auto`
4. T-5ドライラン（成果物生成）  
   `uv run python scripts/ops_t5_dryrun.py --race-date YYYYMMDD --feature-set realtime --fetch-rt --fail-on-empty`
