# ダウンロードリスト v1.2（単勝・T-5、2016〜現在）

この文書は「**2016〜現在を再現可能に学習**するための取得」と、「**運用当日（T-5）**の入力を作る取得」を **用途で分けて定義**します。

* **Backfillable（再現学習に使える）**：2016〜現在を再現可能に学習・検証できるデータ
  * 取得APIは **JVOpen が中心**だが、**0B41（時系列オッズ）は JVRTOpen の racekey 指定でバックフィル**する（詳細は本書 2)）
* **Realtime-only（運用開始以降のみ）**：運用当日の監視/スナップ用途で取得するが、長期バックフィルは前提にしないデータ
  * 例：0B11（WH）, 0B16（WE/AV/JC/TC/CC）, 0B13/0B17（速報DM/TM）

> 重要：**T-60〜T-5の時系列特徴量（傾き/ボラ等）を運用で使うなら、その特徴量で学習している必要がある。**
> したがって **Model M / Model O で 0B41 を使う場合は、学習・検証用にも 0B41 をバックフィル**する（本書 2)）。

## 1) 必須：学習・検証データ構築（JVOpen＝蓄積系）

学習データ（2016〜現在）を再現可能に作るために **最低限これだけ**を落とします。

|  優先 | dataspec（JVOpen）  | 主レコード                  | 使う特徴量（対応）                      | 理由/備考                                            |
| --: | ----------------- | ---------------------- | ------------------------------ | ------------------------------------------------ |
| ★★★ | **DIFF（マスタ差分）**    | UM / KS / CH（他混在あり）     | マスタ（馬/騎手/調教師）                   | 初回ロード推奨。FK回避・欠損補完の前提になることが多い                   |
| ★★★ | **RACE（セットアップ）**  | RA / SE / HR / O1（※）   | 過去走・レース条件・ラベル（勝敗）・確定最終オッズ（＋票数合計） | セットアップデータに RA/SE/HR/O1 が含まれる（長期収録）              |
| ★★★ | **MING（蓄積マイニング）** | DM / TM（直前時点）          | DM/TM特徴量（学習はMINGを正）            | MINGは「直前予想(馬体重発表後)時点」のDM/TMを月曜14:00頃提供           |
| ★★★ | **SNPN（出走時点情報）**  | CK（出走別着度数）             | CKミニ特徴量（着度数・脚質傾向・賞金系など）        | 出走馬名表発表時点の着度数情報として提供。JVOpenでSNPN指定可              |
| ★★☆ | **SLOP（坂路調教）**    | （坂路調教レコード）             | 調教特徴量（坂路）                      | 提供タイミング/単位が記載（毎日、不規則）                            |
| ★★☆ | **WOOD（ウッド調教）**   | WC                     | 調教特徴量（ウッド）                     | WOODは「2021/7/27以降」のウッド調教情報と記載（2016〜2021は欠損前提で設計） |
| ★☆☆ | **COMM（コース情報など）** | CS                     | コース固定情報（トラック/改修等）              | コース情報CSがCOMMに含まれる記載。なくても回るがあると便利                 |

> JVOpenで指定可能な dataspec の一覧に DIFF/RACE/MING/SNPN/SLOP/WOOD/COMM（+YSCH等）が含まれることは仕様書に明記されています。
>
> ※ RACEには環境/時期によりH1等が混在し得ますが、本プロジェクトの最小構成では **O1の「票数合計」**を利用する前提で、H1は必須にしません（必要になったら別途対応）。

### 重要：SNPNについて（あなたの補足を反映）

* **本プロジェクトでは “SNPNのみ” を使用**（SNAPは使用しない）
* あなたが「SNPNで全期間ダウンロードできることを確認済み」なので、2016〜現在は **SNPNで統一取得**でOK

  * 実装は「SNPNだけを取得し、取り込み結果で期間カバレッジを監査（欠損があればアラート）」にします。

## 2) 必須（Model M / Model O 学習・検証）：0B41 バックフィル（JVRTOpen）

0B41（O1時系列）は **JVOpenでは取れず**、JVRTOpen の **racekey 指定**で取得します。
Model M（Market）/ Model O（最終オッズ予測）で 0B41 を特徴量に使うなら、**学習・検証期間（2016〜現在）についても 0B41 を用意**します。

|  優先 | dataspec（JVRTOpen） | レコード           | 使う特徴量（対応）                                | 備考 |
| --: | ------------------ | -------------- | ---------------------------------------- | --- |
| ★★★ | **0B41**           | O1（時系列オッズ：単複枠） | T-5オッズ、オッズ推移（傾き/変化/ボラ等）、最終オッズ予測モデル入力 | **2003年〜現在で取得可能**（参照: `docs/jvlink_handover.md`） |

### 2.1 手順（最低限）

1. 対象レースの **racekey 一覧**を作る（2016〜現在）
   * 推奨：`core.race`（`race_date, track_code, race_no`）から `YYYYMMDDJJRR` 形式を生成して使う
   * 代替：SE等のレースキー（`YYYYMMDDJJKKHHRR`）をそのまま使う
2. `scripts/extract_rt_jvlink.py --dataspec 0B41 --racekey ...` を **racekey ループ**で実行し、JSONLを保存する
3. `scripts/load_to_db.py` で JSONL を投入し、`core.o1_header/core.o1_win` に入ることを確認する

> 実務上はレース数が多いので、「年ごと/開催日ごとに分割」「途中再開」「重複排除（raw.jv_rawのpayload_hash）」を前提にラッパー化する。

| ★★☆ | **0B13**           | DM（速報タイム型）          | 当日推論用DM（kbn=3→2→1）                   | 速報DM 0B13。上書き更新の説明        |

# 5) 「このリストだけで特徴量仕様を満たせる？」チェック

結論、**Backfillable特徴量（2016〜再現可）** は満たせます（Model FはJVOpen最小セット、Model M/Oは 0B41バックフィル込み）。
一方で **Realtime-only（0B11/0B16/0B13/0B17）** は、運用当日は満たせますが、学習・検証での扱いは別途設計が必要です。

### Backfillable（2016〜再現可：JVOpenで構築）

* **近走成績・集約・自作SR**：RACE（RA/SE）で過去走と条件が揃う
* **ラベル（1着）**：SE/HR（結果・払戻）で確定可能
* **確定最終オッズ（教師・EV検証）**：セットアップO1（確定オッズ）
* **DM/TM（学習）**：MING（直前時点の確定スナップ）
* **CK**：SNPN（CK）
* **馬体重（学習のフォールバック）**：SE内の `body_weight/body_weight_diff`（存在する場合）
* **時系列オッズ（Model M/O 学習）**：0B41（O1時系列）のバックフィル（本書 2)）

### Realtime-only（運用当日：JVRTOpenで取得）

* **馬体重（運用T-5）**：0B11（WH）
* **変更・取消フラグ（運用T-5）**：0B16（WE/AV/JC/TC/CC）
* **DM/TM（運用T-5）**：0B13/0B17（kbn=3→2→1）

> したがって、学習期間が 2016〜現在でも、
>
> * **Model F（オッズなし）**：Backfillableで全期間（2016〜）を再現学習できる
> * **Model M / Model O（0B41依存）**：0B41バックフィルにより、同様に **2016〜現在**で学習・検証できる
>
> という「二階建て」が前提になります。
