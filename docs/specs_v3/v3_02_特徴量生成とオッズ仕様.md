# v3 特徴量生成とオッズ仕様

> **スコープ**: `build_features_v3.py`, `odds_v3_common.py`

---

## 1. 入力

- 主入力: `data/features_v2.parquet`
- 必須列: `race_id`, `horse_no`, `target_label`, `race_date`, `field_size`

---

## 2. v3 で追加する列

| 列名 | 定義 | 型 |
|---|---|---|
| `y_win` | `target_label == 3`（1着） | int |
| `y_place` | `target_label >= 1`（3着以内） | int |
| `finish_pos` | DB `core.result` から結合 | float (NaN許容) |
| `odds_win_final` | final スナップショットの単勝オッズ | float |
| `p_win_odds_final_raw` | `1 / odds_win_final` | float |
| `p_win_odds_final_norm` | レース内正規化した暗示確率 | float |
| `odds_win_t10` | t10 スナップショットの単勝オッズ | float |
| `p_win_odds_t10_raw` | `1 / odds_win_t10` | float |
| `p_win_odds_t10_norm` | レース内正規化した暗示確率 | float |
| `odds_final_data_kbn` | final snapshot の `data_kbn` | int |
| `odds_t10_data_kbn` | t10 snapshot の `data_kbn` | int |
| `odds_final_announce_dt` | final の発表日時 | datetime |
| `odds_t10_announce_dt` | t10 の発表日時 | datetime |
| `odds_t10_asof_dt` | t10 の基準時点（`race_start - 10min`） | datetime |

---

## 3. オッズスナップショット方針

### 3.1 final（検証用）

- `data_kbn` 優先順位: `4 > 3 > 2 > 1`
- `announce <= race_start` の範囲で最新を採用
- 用途: 検証、校正比較、Benter R* 比較

### 3.2 t10（運用想定）

- `as_of = race_start - 10min` を基準時点とする
- `announce <= as_of` の範囲で最新を採用
- 用途: 運用推論、default の binary / PL 学習

### 3.3 欠損処理

- オッズが `NULL / 0 / 負` の場合は欠損扱い
- レース内で再正規化する
- レース内で全欠損なら `NaN`

---

## 4. リーク防止アサーション

- `assert_t10_no_future_reference()` により、t10 の `announce_dt <= asof_dt` を全行で検証する
- v3 の as-of 整合性は最優先事項として扱う

---

## 5. メタデータ出力

`features_v3_meta.json` に以下を記録する。

- 入出力パス、行数、レース数、列一覧
- カバレッジ（`finish_pos`, odds 各列の非 NaN 率）
- `operational_default = "t10_only"`
- `contains_final_odds_columns`
- `contains_t10_odds_columns`
- コードハッシュ（`build_features_v3.py` + `odds_v3_common.py`）

---

## 6. feature table と feature contract の分離

- `features_v3` には final / t10 odds 列を保持してよい
- ただし、学習で実際に使う列は `scripts_v3/feature_registry_v3.py` の contract で制御する
- 列が `features_v3` に存在することと、学習に投入してよいことは同義ではない

この分離により、以下を抑制する。

- train-serve mismatch
- accidental leakage
- entity ID 依存
- PL 入力の無制限な肥大化
