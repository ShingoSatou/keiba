# 実装ギャップ/ToDo（仕様 ↔ 実装 ↔ DB）v0.2（DB作成以降）

* 対象プロジェクト: `keiba`
* 前提: **DB作成（DDL適用）自体は別工程**。この文書は「DBが作成された後」にやるべき工程（ETL→特徴量→学習→運用）のタスクを、仕様↔実装のギャップとして整理する
* 目的: 仕様書に書かれている要件に対して、実装（パーサ/ETL/DDL/運用フロー/ML）で未整備・不整合になっている点を「直す順番付き」で管理する
* 方針: **この文書は“実装を直すためのチェックリスト”**。仕様（何をしたいか）は各仕様書を正とする
* 最終更新: 2026-02-14

## 0. 前提（用語 / 正とする仕様）

* `dataspec`：JVOpen/JVRTOpenに指定するデータ種別（例: `RACE`, `DIFF`, `MING`, `SNPN`, `0B41`）
* `rec_id`：レコード種別（payload先頭2文字。例: `RA`, `SE`, `O1`, `CK`, `DM`, `TM`）
* 優先度:
  * **P0**: ブロッカー（落ちる/汚染する/検証が成立しない/運用で事故る）
  * **P1**: 重要（動くが価値が出ない、または運用上危険）
  * **P2**: 低優先（仕様上はあるが後回しでよい）
* DB一次ソース（設計・DDL）: `init_db.sql`, `init_mart.sql`, `docs/migrations/*`
* 実装一次ソース:
  * パーサ: `app/infrastructure/parsers.py`
  * ローダ（JSONL→DB）: `scripts/load_to_db.py`
  * 特徴量生成（core→mart）: `scripts/build_features.py`
  * データセット生成（mart→parquet）: `scripts/build_dataset.py`
  * 学習/評価: `scripts/train.py`, `scripts/backtest.py`
  * 当日運用（取得）: `scripts/extract_rt_jvlink.py`（※Windows/32bit前提）
* 正とする仕様書（参照）:
  * `docs/全体仕様書.md`（要件・T-5・モデル構成）
  * `docs/データ取得仕様書.md`（取得→JSONLの責務境界）
  * `docs/ETL・パイプライン仕様書.md`（JSONL→DB投入）
  * `docs/データ辞書・正規化仕様書.md`（レコード→カラム対応）
  * `docs/特徴量仕様.md` / `docs/特徴量カタログ.md`（特徴量）
  * `docs/検証設計仕様書.md`（walk-forward/リーク防止/stacking）
  * `docs/運用仕様書.md`（当日T-5運用・監査ログ）

> 注意（重要）：特徴量は **Backfillable（2016〜再現可）** と **Realtime-only（運用開始日以降のみ）** に分かれる。**0B41はracekey指定でバックフィル可能**だが、0B11/0B16/0B13/0B17 依存の入力は、現時点では運用開始日以降のみ（参照: `docs/ダウンロードリスト.md`, `docs/特徴量カタログ.md`）。

```text
DB作成（別工程）
  init_db.sql / init_mart.sql / docs/migrations/*
    ↓
ETL（JSONL→DB）
  data/*.jsonl → raw/core（scripts/load_to_db.py）
    ↓
特徴量（core→mart）
  mart.run_index/horse_stats/person_stats 等（scripts/build_features.py）
  + T-5スナップ（Backfillable/Realtime-only）（※未整備: TODO）
    ↓
学習・検証
  mart → data/train.parquet（scripts/build_dataset.py）
  → train/backtest（scripts/train.py, scripts/backtest.py）
    - Model F: Backfillable（2016〜）
    - Model M / O: Backfillable（0B41バックフィル込みで2016〜。未バックフィルなら運用開始日以降のみ）

運用（当日T-5）
  JVRTOpen(0B41/0B11/0B16/0B13/0B17) → JSONL → DB
  → T-5スナップ確定 → 推論 → 出力/監査（※未整備: TODO）
```

### 2.3 特徴量: **T-5スナップ（as-of）をDB上で再現可能にする（学習/運用の土台）**

仕様上の最重要（参照: `docs/全体仕様書.md`, `docs/運用仕様書.md`, `docs/検証設計仕様書.md`）:

* 入力時点は **T-5**（`asof_ts = post_time - 5min`）
* すべての入力は `<= asof_ts` の最新を採用し、採用キーを監査ログに残す

補足（二階建て）:

* Backfillable（2016〜）には2系統ある（参照: `docs/ダウンロードリスト.md`）
  * Model F（JVOpenのみ）: **0Bxx由来の採用キーが存在しない**（NULLになり得る）
  * Model M/O（0B41バックフィル込み）: `o1_announce_mmddhhmi` 等が存在する（監査に残す）
* Realtime-only（運用開始以降）では **採用キーが監査に必須**（WH/変更/速報DM/TM 等）

現状（実装）:

* `mart` には `T-5スナップ` のテーブルが存在しない（`init_mart.sql` は run_index/horse_stats/person_stats/prediction_log のみ）
* `scripts/predict.py` は「レースID + 手入力オッズ(10分前)」で推論するのみ（as-of/T-5前提ではない）

チェック:

* [ ] `mart` に **T-5スナップ（1行=1頭）** のテーブルを定義する（例: `mart.t5_runner_snapshot`）
  * [ ] キー: `race_id, horse_id(or horse_no), asof_ts`
  * [ ] `feature_set`（例: `backfillable` / `realtime`）を持たせる（またはスナップ表を分ける）
  * [ ] 採用キー（監査に必須）: `o1_announce_mmddhhmi`, `wh_announce_mmddhhmi`, `dm/tm_kbn`, `dm/tm_create_time`, `event_change_id(or announce)` 等
  * [ ] 運用の監査ログに必要な列（モデル/特徴量バージョン、τなど）
* [ ] `core` から `mart.t5_*` を生成するバッチ（またはSQLビュー）を用意する
  * [ ] `TC` を反映して post_time を更新し、asof_ts を再計算できる
  * [ ] `AV` を反映して除外（`scratch_flag` 反映 or snapshot生成で除外）
  * [ ] `JC` を反映して jockey を置換し、フラグを立てる

受け入れ条件:

* [ ] 「同一レースで複数データ更新」があっても、T-5時点の入力が再現できる（採用キーが残る）
