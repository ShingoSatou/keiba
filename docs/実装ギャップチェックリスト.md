# 実装ギャップチェックリスト（仕様 ↔ 実装 ↔ DB）v0.1

* 対象プロジェクト: `keiba`
* 目的: 仕様書に書かれている要件に対して、実装（パーサ/ETL/DDL/運用フロー）で未整備・不整合になっている点を「直す順番付き」で管理する
* 方針: **この文書は“実装を直すためのチェックリスト”**。仕様（何をしたいか）は各仕様書を正とする
* 最終更新: 2026-02-13

---

## 0. 前提（用語）

* `dataspec`：JVOpen/JVRTOpenに指定するデータ種別（例: `RACE`, `DIFF`, `MING`, `SNPN`, `0B41`）
* `rec_id`：レコード種別（payload先頭2文字。例: `RA`, `SE`, `O1`, `CK`, `DM`, `TM`）
* DB一次ソース: `init_db.sql`
* パーサ一次ソース: `app/infrastructure/parsers.py`
* ローダ一次ソース: `scripts/load_to_db.py`

---

## 1. ブロッカー（最優先：ETLが落ちる／データが汚染される）

### 1.1 `core.runner` のスキーマ差分（DDL ↔ ローダ）

現状、ローダが存在しない列へINSERT/UPDATEしようとして落ちうる。

* 参照
  * DDL: `init_db.sql` の `core.runner`
  * ローダ: `scripts/load_to_db.py` の `upsert_runner()`（`runner_sql`）
  * パーサ: `app/infrastructure/parsers.py` の `RunnerRecord`

チェック:

* [ ] 「どちらを正にするか」を決める（推奨: **DDL/設計を正**にして不足列が必要ならDDLへ追加）
* [ ] DDL側を更新する場合
  * [ ] `core.runner` に必要列を追加（例: `data_kubun`, `trainer_code_raw`, `trainer_name_abbr`, `jockey_code_raw`, `jockey_name_abbr` 等）
  * [ ] `docs/DB設計仕様書.md` と `docs/データ辞書・正規化仕様書.md` の対応表も同時更新
* [ ] ローダ側を更新する場合
  * [ ] `scripts/load_to_db.py` のINSERT列をDDLに合わせる（余計な列を消す）
  * [ ] `RunnerRecord` のフィールド/利用箇所も合わせて整理

受け入れ条件:

* [ ] `uv run python scripts/load_to_db.py --input-dir data/` が `core.runner` INSERTで落ちない

### 1.2 `0B41`（時系列オッズ）を `core.o1_*` へ投入するルーティング

現状 `rec_id="O1"` を一律「最終オッズ」として処理すると、`0B41`（時系列）を `core.odds_final` に誤投入するリスクがある。

* 参照
  * パーサ: `OddsTimeSeriesRecord.parse`（`app/infrastructure/parsers.py`）
  * ローダ: `upsert_o1_timeseries()`（`scripts/load_to_db.py`）

チェック:

* [ ] ローダで `dataspec` を見て分岐できるようにする（例: ファイル先頭が `0B41_...` なら時系列扱い）
* [ ] `dataspec=0B41` の `rec_id="O1"` は `OddsTimeSeriesRecord.parse` → `upsert_o1_timeseries` に流す
* [ ] `dataspec=RACE` の `rec_id="O1"` は `OddsRecord.parse` → `upsert_odds`（= `core.odds_final`）に流す
* [ ] ルーティングが曖昧な場合に備え、ガード（警告/スキップ）を入れる（例: `announce_mmddhhmi` が有効なら時系列扱い、等）

受け入れ条件:

* [ ] `0B41_*.jsonl` をロードしても `core.odds_final` が増えない（増えるのは `core.o1_header/core.o1_win` のみ）

## 2. 重要（動くが価値が出ない／運用上危険になり得る）

### 2.1 DM/TM（`MING` と `0B13/0B17`）の正規化不足

現状 `horse_no=0` 固定のため、(race_id, horse_no) PKで衝突・上書きが起きる想定。

* 参照
  * DDL: `core.mining_dm`, `core.mining_tm`（`init_db.sql`）
  * パーサ: `DMRecord`, `TMRecord`（`app/infrastructure/parsers.py`）
  * ローダ: `insert_mining_record()`（`scripts/load_to_db.py`）
  * 仕様: `docs/全体仕様書.md` / `docs/運用仕様書.md`（学習はMING、運用は速報でT-5採用）

チェック:

* [ ] `horse_no` を正しく抽出する（1頭単位に正規化できること）
* [ ] DM/TMの主要値（スコア/順位/予想タイム等）を `core.mining_*` の列へ入れる
* [ ] 「複数回発表」の扱いを決める
  * [ ] `core.mining_*` を「採用スナップのみ」格納にする（=常に上書き）か
  * [ ] 時系列で持つならスキーマ側を拡張する（announce/create等をキーへ）

受け入れ条件:

* [ ] `MING_*.jsonl` をロードしてもPK衝突で上書きされない（or 意図通りの上書きになる）

* [ ] `data_create_ymd` / `announce_mmddhhmi` をパースしてDBへ入れる（監査キー）
* [ ] 主要イベントを「運用ルール」に接続する
