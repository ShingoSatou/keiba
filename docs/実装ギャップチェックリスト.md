## 0. 前提（用語 / 正とする仕様）

* `dataspec`：JVOpen/JVRTOpenに指定するデータ種別（例: `RACE`, `DIFF`, `MING`, `SNPN`, `0B41`）
* `rec_id`：レコード種別（payload先頭2文字。例: `RA`, `SE`, `O1`, `CK`, `DM`, `TM`）
* 優先度:
  * **P0**: ブロッカー（落ちる/汚染する/検証が成立しない/運用で事故る）
  * **P1**: 重要（動くが価値が出ない、または運用上危険）
  * **P2**: 低優先（仕様上はあるが後回しでよい）
* DB一次ソース（設計・DDL）: `migrations/*.sql`（bootstrap: `migrations/0001_init_db.sql`, `migrations/0002_init_mart.sql`）
* 実装一次ソース:
  * パーサ: `app/infrastructure/parsers.py`
  * ローダ（JSONL→DB）: `scripts/load_to_db.py`
  * 特徴量生成（core→mart）: `scripts/build_features.py`
  * データセット生成（mart→parquet）: `scripts/build_dataset.py`
  * 学習/評価: `scripts/train.py`, `scripts/backtest.py`
  * 当日運用（取得）: `scripts/extract_rt_jvlink.py`（※Windows/32bit前提）
* 実装特徴量（現状の列/算出方法）: `docs/実装特徴量一覧.md`（仕様との差分棚卸し用）
* 正とする仕様書（参照）:
  * `docs/全体仕様書.md`（要件・T-5・モデル構成）
  * `docs/データ取得仕様書.md`（取得→JSONLの責務境界）
  * `docs/ETL・パイプライン仕様書.md`（JSONL→DB投入）
  * `docs/データ辞書・正規化仕様書.md`（レコード→カラム対応）
  * `docs/特徴量仕様.md` / `docs/特徴量カタログ.md`（特徴量）
  * `docs/検証設計仕様書.md`（walk-forward/リーク防止/stacking）
  * `docs/運用仕様書.md`（当日T-5運用・監査ログ）

> 注意（重要）：特徴量は **Backfillable（2016〜再現可）** と **Realtime-only（運用開始日以降のみ）** に分かれる。**0B41はracekey指定でバックフィル可能**だが、0B11/0B14/0B13/0B17 依存の入力は、現時点では運用開始日以降のみ（参照: `docs/ダウンロードリスト.md`, `docs/特徴量カタログ.md`）。

## 1. 工程マップ（DB作成以降）

```text
DB作成（別工程）
  migrations/*.sql
    ↓
ETL（JSONL→DB）
  data/*.jsonl → raw/core（scripts/load_to_db.py）
    ↓
特徴量（core→mart）
  mart.run_index/horse_stats/person_stats 等（scripts/build_features.py）
  + T-5スナップ（mart.t5_runner_snapshot; scripts/build_t5_snapshot.py）
    ↓
学習・検証
  mart → data/train.parquet（scripts/build_dataset.py）
  → train/backtest（scripts/train.py, scripts/backtest.py）
    - Model F: Backfillable（2016〜）
    - Model M / O: Backfillable（0B41バックフィル込みで2016〜。未バックフィルなら運用開始日以降のみ）
    ↓
運用（当日T-5）
  JVRTOpen(0B41/0B14/0B11/0B13/0B17) → JSONL → DB
  → T-5スナップ確定 → 推論 → 出力/監査（※未整備: TODO）
```

### 2.5 運用: 当日T-5推論フロー（取得→DB→T-5スナップ→推論→出力→監査）

現状（実装）:

* 取得（JVRTOpen→JSONL）: `scripts/extract_rt_jvlink.py`（単発）/ `scripts/ops_rt.py`（当日一括・dataspecごとのキー単位）
* 当日RACE補完: `scripts/ops_race.py`（RACE抽出→DB投入）
* RT投入分離: `scripts/ops_rt_load.py`（dataspec別ポリシー + 投入件数サマリ）
* JSONL→DB→特徴量→T-5スナップ→推論→出力: `scripts/ops_t5_dryrun.py`（`load_to_db`→`build_features`→`build_t5_snapshot`→`predict_t5`→CSV/JSON/HTML + 監査JSON）
* `scripts/predict_t5.py` が `mart.t5_runner_snapshot` 入力の推論（単一モデル）と監査ログ出力を担当
* `scripts/ops_t5_dryrun.py` は `--fetch-rt-policy auto`（dataspec別: `0B14=strict`, `0B41/0B11=warn`, `0B13/0B17=continue`）を持ち、停止/継続を自動判定する
* `scripts/ops_t5_dryrun.py` は `--ensure-race`（デフォルトON）で `core.race/core.runner` 前提不足時に `ops_race.py` を実行できる
* `scripts/build_t5_snapshot.py` は `--ensure-race`（デフォルトON）と `missing_start_time` stop/warn 判定（`stop>=0.50`, `warn>0`）を持ち、前提不足日を `ops_race.py` で補完→再判定できる
* `scripts/ops_t5_dryrun.py` は `--fail-on-empty` を持ち、`t5_snapshot rows=0` を運用失敗（非0終了）として扱える
* `scripts/ops_rt.py` は dataspecごとにキー単位を切り替える（`0B41` はレース単位、`0B14/0B11/0B13/0B17` は開催日単位）
* 実行例: `uv run python scripts/ops_t5_dryrun.py --race-date YYYYMMDD --feature-set realtime`（成果物: `data/ops/<YYYYMMDD>/`）

参照:

* 仕様: `docs/運用仕様書.md`（取得タイミング、スナップ採用、出力、監査）
