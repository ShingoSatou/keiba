# データ品質レポート（学習/テスト：`data/train.parquet`）

最終更新: 2026-02-20

この文書は、学習・評価に使う `data/train.parquet` の品質を点検し、発見された不良と対応状況を整理する。

## 1. 前提（どのデータが学習に使われるか）

* `data/train.parquet` は `scripts/build_dataset.py` が DB（core/mart）から生成する（`--output` のデフォルト）。
* 学習は DB を直接見ず、`scripts/train.py` が `--input`（デフォルト `data/train.parquet`）を読み込む。

## 2. 再現手順（品質チェック）

```bash
# (任意) core.race の不良をRAから再投入（不良候補のみ）
uv run python scripts/backfill_core_race.py --from-date 2016-01-01 --only-bad

# mart 再構築（2016+）
uv run python scripts/build_features.py --rebuild --from-date 2016-01-01

# train.parquet 再生成（2016+）
uv run python scripts/build_dataset.py --from-date 2016-01-01 --min-horses 5 --output data/train.parquet

# 品質チェック（学習で使う特徴量＝FEATURE_COLSを中心に集計/ゲート判定）
uv run python scripts/check_dataset_quality.py --input data/train.parquet --gate --json-output data/quality/train_latest.json
```

## 4. 旧データで発見した重大不良（参考: 2026-02-08 時点の観測）

※当時の `data/train.parquet` は stub/placeholder を大量に含んでおり、現在は再生成により置き換え済み。

### P0（モデル評価以前に修正必須）→ 対応済み

1) **レース条件が stub/placeholder のまま混入**

* `surface==0` / `distance_m==0` 行が大半 → 学習データが成立しない
* 対応: `scripts/build_dataset.py` と `scripts/build_features.py` の抽出条件を強化し、stub を除外。

2) **ラベル整合性が崩れたレースが混入（勝ち馬なし、少頭数など）**

* `sum(is_win)==0` のレース、`<5頭` レースなどが混在
* 対応: `scripts/build_dataset.py` で「勝ち馬(=finish_pos=1)が存在するレースのみ」「`--min-horses`」を適用。

3) **特徴量のバグ/無効化**

* `distance_change_m` が全行 0（定数特徴量）等
* 対応: `scripts/build_dataset.py` の当日コンディション計算を修正。

4) **Parquet と DB の不整合（生成時点/生成元が混在）**

* Parquet 上のレース条件が DB と一致しない
* 対応: `scripts/build_dataset.py` による再生成を正とし、品質ゲートで破綻を検知。

## 5. 改善（投入済みの修正）

### 5.1 `scripts/build_dataset.py` の修正

* **不完全なレースを除外**（stub/placeholder の混入防止）:
  * `surface IN (1,2)`（デフォルト。`--include-obstacles` で (1,2,3)）/ `distance_m > 0` / `finish_pos=1 が存在するレースのみ` を採用。
* **`distance_change_m` を前走距離ベースで計算**するように修正（定数化バグの解消）。

### 5.2 品質チェック用スクリプト追加

* `scripts/check_dataset_quality.py` を追加。
  * 分割ごとの件数（行/レース/陽性数）、winner数整合性、placeholder率、
    欠損・定数特徴量、主要アノマリ（`going==0` 等）を出力する。
  * `--gate` で品質基準違反時に `exit 1` で停止できるようにした。
  * **未使用特徴量は調査対象から外す**（学習入力 `FEATURE_COLS` を中心に集計/ゲート判定する）。

### 5.3 RAパース/投入の修正

* `app/infrastructure/parsers.py`
  * `distance_m` の異常スケール（`0<d<100`）を `*100` 補正。
  * `going/weather/field_size` の `0` を欠損（`None`）として扱う。
  * `field_size` は `RA_STARTERS`（出走頭数）を優先して採用。
* `scripts/load_to_db.py` / `app/services/ingest_service.py`
  * `upsert_race` の `ON CONFLICT` を更新し、`distance_m<800` の既存値も正値で上書き可能にした。
  * `class_code` を conflict 時に `NULL→0` で統一するよう修正。

### 5.4 特徴量/学習データ再生成の修正

* `scripts/build_features.py`
  * `--from-date/--to-date` を追加（デフォルト開始日: `2016-01-01`）。
  * `surface IN (1,2)`（デフォルト。`--include-obstacles` で (1,2,3)）かつ `distance_m>0` のレースのみで `mart.*` を計算。
* `scripts/build_dataset.py`
  * `--from-date` のデフォルトを `2016-01-01` に変更。
  * `--min-horses`（デフォルト5）を追加し、少頭数レースを除外可能にした。
  * `class_code` を `NULL→0` で統一し、`going/field_size` の `0` を欠損化。

### 5.5 ベースライン学習の入力特徴量の見直し（欠損率対策）

* `wood_*` はデータ期間の都合で欠損率が高いため、ベースライン学習の入力から一時除外。
* `*_sim_*` / `n_sim_runs_5` も欠損率が高いため、ベースライン学習の入力から一時除外。

## 6. 現時点の残課題（2026-02-20 時点）

### P0（ゲートPASSだが、モデル評価前に解消推奨）

1) **`class_code` が定数（常に0）**

* 現状 `RaceRecord.parse` は `class_code = 0` 固定（未実装）で、学習上は情報量ゼロ。
* 対応方針（要選択）:
  * (A) 当面 `class_code` を学習入力から外す
  * (B) 取得元（RA以外レコード/別テーブル）を特定して backfill する

2) **障害レース（`surface=3`）を今回の予測対象から外している**

* 傾向やデータ定義が異なるため、ベースラインは芝/ダ専用とする。
* 必要になったら「障害専用モデル」または「障害を含めた別学習設定」を用意する。

### P1（仕様上の欠損が多いが、説明は必要）

* `speed_std_5` / `speed_trend_3` の欠損（約21.9%）は、過去走が少ない馬で定義上発生する。
* 坂路（SLOP）系は欠損（約11.4%）がある前提で、missing_flag と併用する。

### P2（観測メモ）

* `n_runs_5` 欠損は「`going` 欠損時の `going_bucket` 判定差」により結合できていなかったのが原因で、`scripts/build_dataset.py` の `going_to_bucket` を修正して解消（`pd.isna()` を考慮）。
* `ck_missing_flag` / `ck_h_dirt_straight_starts` が定数になっているため、モデル入力としては効果が薄い（必要なら入力から除外）。
