## 1. 前提（確定した意思決定）

- **対象馬券種:** ワイド
- **問題設定:** Listwise（Learning-to-Rank）
- **評価指標（第1段）:** NDCG@3（`docs/specs_v2/競馬予測システム 課題設定・要件定義書.md`）
- **MVP範囲:** PoC（学習 → OOF評価 → バックテスト → 可視化）
  - 発走5分前（T-5）推論、オッズ・ドロップ予測、自動投票は **次フェーズ**
- **DB方針:** 既存DBに触れず、v2専用DB `keiba_v2` を作成して利用
  - スキーマは `raw/core/mart`（DDL一次ソースは `migrations_v2/*.sql`）
- **オッズ方針（PoC）:** まずは **確定オッズ（Oracle）**で検証する
  - ワイド確定オッズ: `O3`（`core.o3_wide` の `min_odds_x10` を保守的に実効オッズとして使用）
  - 決済: `HR` のワイド払戻（`core.payout` の `bet_type=5`）
- **初期フェーズの対象レース（セグメント）:**
  - 中央競馬（場コード01〜10）のみ（取り込み段階で除外）
  - ダートのみ / 障害除外 / 芝除外
  - 4歳以上の条件戦 + オープン・重賞（新馬/未勝利は除外）
  - ※絞り込みの実装は **Phase 2（特徴量生成）** で行う（Phase 1は「取り込みの網羅」を優先）
- **リポジトリ方針:** v1は参照のみ（編集しない）
  - v2実装: `migrations_v2/`, `scripts_v2/`, `test_v2/`, `app/`, `frontend/`

## 3. MVPのゴール（成功条件）

最低限、以下が「再現性あり・リークなし」で回ること。

1. `keiba_v2` に `raw/core` を構築し、JV-Data（RA/SE/HR/O1/O3/WH/WE/DM/TM…）を投入できる
2. `data/features_v2.parquet`（**1行=1頭、Race_ID昇順ソート**）を生成できる
3. Listwise Ranker（LightGBM LambdaMART）を **Walk-forward + Race_IDグループ分割**で学習し、OOF予測を出せる
4. OOFから Platt Scaling（ロジスティック回帰）で **Top3確率**に校正できる
5. Top3確率から **Plackett–Luce + MonteCarlo** でワイド同時確率 `P(i&j in top3)` を推定できる（ベースライン）
6. `O3(min_odds)` と `HR(payout)` を用いて EV を計算し、fractional Kelly + 1レース上限で資金配分しバックテストできる
7. `data/backtest_result.json`（UI互換）を生成し、`frontend` で閲覧できる

## 4. 全体アーキテクチャ（MVP）

### 4.1 データフロー

1) 取り込み: JV-Data → JSONL → v2 DB（`raw.jv_raw` / `core.*`）  
2) 特徴量生成: `core.*` → `data/features_v2.parquet`  
3) 学習（第1段）: features → Ranker（`LGBMRanker`）→ `data/oof/ranker_oof.parquet`  
4) 校正（第2段）: ranker OOF → Platt（`LogisticRegression`）→ `data/oof/top3_oof.parquet`  
5) ワイド確率（第3段・ベースライン）: top3_oof → PL(MC) → ペア確率  
6) 期待値/資金配分/バックテスト: ペア確率 + O3(min_odds) + HR(payout) → `data/backtest_result.json`  
7) 可視化: FastAPI `/ui/*` + Vite UI

### 4.2 技術選定（固定）

- ランキング学習: LightGBM LambdaMART（`LGBMRanker`, `eval_at=[3]`）
- 校正: `LogisticRegression(class_weight="balanced")`
- ワイド同時確率: Plackett–Luce + Monte Carlo（PoCは 2,000〜10,000 回/レースで調整）
- 資金配分: fractional Kelly + 1レース上限（暴発防止）
- オッズ（PoC）: `O3` の `min_odds_x10` を実効オッズとして使用（保守的）

### Phase 2: 特徴量生成（`core` → `data/features_v2.parquet`）

**目的**
- v2 PoC の学習入力（1行=1頭）を、時系列リークなしで再現可能に生成する
- 初期フェーズ対象セグメント（中央×ダート×4歳以上×新馬/未勝利除外）を **Phase 2 で固定実装** する

**成果物**
- `scripts_v2/build_features_v2.py`
- `data/features_v2.parquet`
- `data/features_v2_meta.json`（期間/件数/列/seed/コードhash）
- `test_v2/test_features_v2_no_leakage.py`（as-of検証）

**Phase 2先頭で実施する前提整備（必須）**
- `app/infrastructure/parsers.py`（RA/SE）を拡張し、特徴量抽出条件に必要なコードを投入可能にする
  - RA: `race_type_code`, `weight_type_code`, `condition_code_min_age`（必要に応じて `class_code` も再定義）
  - SE: `sex`（性別コード）
- `scripts_v2/load_to_db.py` の upsert を上記列対応に更新
- 既存データの backfill を実施（同一JSONL再投入で冪等upsert）

**入力（DB）**
- `core.race`, `core.runner`, `core.result`
- `core.horse`, `core.jockey`, `core.trainer`
- `core.mining_dm`, `core.mining_tm`, `core.rt_mining_dm`, `core.rt_mining_tm`（Group E）
- ※特徴量に **確定オッズ/最終人気**は使用禁止（仕様書 4.2）

**抽出条件（初期フェーズのセグメント / 実装固定）**
- 中央競馬（track_code 01-10）
- `surface=2`（ダートのみ）
- 障害除外（`surface=3`）/ 未知除外（`surface=0`）
- 4歳以上: `race_type_code=14`
- 新馬/未出走/未勝利除外: `condition_code_min_age NOT IN (701, 702, 703)`
- 条件戦 + OP/重賞を主対象とするため、`condition_code_min_age IN (5, 10, 16, 999)` を基本採用
- 品質条件: `distance_m > 0`, `field_size > 0`, `horse_no BETWEEN 1 AND 18`, `finish_pos IS NOT NULL`

**出力仕様（最低限）**
- 1行=1頭（Race_ID=Query ID で group）
- `race_id` 昇順（必須）、同一レース内は `horse_no` 昇順
- ラベル: `target_label`（1着=3,2着=2,3着=1,4着以下=0）
- 欠損は原則 NaN のまま（ツリーの欠損分岐に委ねる）
- Group A の `age` は `horse_id` 先頭4桁（生年）から算出、`sex` は SE由来を使用（未取得はNaN）

**特徴量（仕様書A〜FのMVP実装）**
- Group A（基礎）: age/sex/斤量/距離/馬場/乗替/騎手/調教師
- Group B（過去走Lag1〜3）: speed_index, up3_index, distance_diff, course_type_match, rank, days_since_lag1
- Group C（適性）: 同距離/同条件の複勝率（過去2年、ベイズ平滑）
- Group D（レース内相対）: Z-score / rank / percentile（★最重要）
- Group E（公式予測）: DM/TM の予測値・順位（および相対Z）
- Group F（陣営）: 騎手/調教師の直近6ヶ月複勝率（前日まで、ローリング集計+平滑）

**Group B指数（speed_index / up3_index）の実装方針（仕様書準拠）**
- 目的: 絶対タイムを「距離補正 + 当日馬場差補正」した比較可能指標へ変換する
- 式（共通）:
  - `distance_correction = 1440 / distance_m`
  - `speed_index = (base_time_sec - time_sec) * distance_correction + 80`
  - `up3_index = (base_3f_sec - final3f_sec) * distance_correction + 80`
- `base_time_sec` / `base_3f_sec` の算出:
  - 過去2年ローリングで基準値を計算（`closed='left'`）
  - Fine（track/surface/distance_bucket/going_bucket）と Coarse（surface/distance_bucket）をベイズ平滑で縮約
  - 当日馬場差は同日同場同surfaceの逐次平均（同日未来レースは参照しない）で補正

**リーク防止（必須）**
- as-of制約: 集計/lagは必ず `t_race` より過去のみ
- 当日情報の逐次制約: 同日内も「対象レースより前に確定したレース」のみ参照
- 自動検証:
  - 集計参照元が `race_datetime < 対象race_datetime` を満たすこと
  - `race_id` 昇順 / レース内 `horse_no` 昇順をテストで保証（失敗時は exit 1）

**DoD**
- `uv run python scripts_v2/build_features_v2.py --from-date 2016-01-01 --to-date 2024-12-31` が完走する
- `data/features_v2.parquet` が仕様（1行=1頭, ソート, `target_label`）を満たす
- `data/features_v2_meta.json` に期間/件数/列/ハッシュ/主要パラメータが保存される
- `uv run pytest -q test_v2/test_features_v2_no_leakage.py` が通る

### Phase 3: Ranker学習（LambdaMART）+ OOF保存

**成果物**
- `scripts_v2/train_ranker_v2.py`
- `scripts_v2/diagnose_ranker_generalization_v2.py`（汎化ギャップ診断）
- `data/oof/ranker_oof.parquet`
- `data/oof/ranker_cv_metrics.json`
- `models/ranker_lgbm.txt`
- `models/ranker_lgbm_all_years.txt`（比較用）
- `models/ranker_bundle_meta.json`

**Split（`docs/specs_v2/時系列交差検証...` 準拠）**
- Walk-forward（年単位、Rolling Windowを本命採用）
- 既定: `train_window_years=5`（例: Train=2016-2020, Valid=2021 → ... → Train=2019-2023, Valid=2024）
- Expanding Window（`inf`）は比較対象として扱う（探索は後続フェーズ）
- 分割単位は Race_ID（同一レースがTrain/Validに跨らない）
- Hold-out: 2025年を封印（実装は `year>=2025` を除外）

**Model（初期値）**
- `LGBMRanker(objective="lambdarank", eval_at=[3])`
- seed固定、early stopping、foldごとに NDCG@3 を記録
- 最終モデルは2本保存（主モデル=直近窓、比較モデル=全期間）

**ハイパーパラメータ最適化（Optuna, Phase 3+）**
- `scripts_v2/tune_ranker_optuna_v2.py` で、Rolling CV（既定 `train_window_years=5`）の平均NDCG@3を最大化するように探索
- 探索対象:
  - `feature_set`: `base`（`data/features_v2.parquet`） / `te`（`data/features_v2_te.parquet`）
  - LightGBM主要params（learning_rate/num_leaves/min_child_samples/reg_alpha/reg_lambda/feature_fraction/bagging_fraction/bagging_freq）
- `data/features_v2_te.parquet` は `scripts_v2/build_features_v2.py --with-te --output data/features_v2_te.parquet ...` で生成する（baseは `--with-te` なし）
- `trainer_key/jockey_key` の生IDは**常に除外**（探索対象にしない）
- 成果物:
  - `data/oof/ranker_optuna_trials.parquet`
  - `data/oof/ranker_optuna_best.json`
  - `data/oof/ranker_optuna_best_params.json`（`train_ranker_v2.py --params-json` 用）
  - `data/optuna/ranker_optuna.sqlite3`（resume用）
- 依存: `uv sync --extra optuna`

**追加モデル（スタッキング用, Phase 3+）**
- XGBoost / CatBoost の Ranker を Optuna で最適化し、比較用のbest paramsを保存する
  - `scripts_v2/tune_ranker_xgb_optuna_v2.py`
  - `scripts_v2/tune_ranker_cat_optuna_v2.py`
  - 出力: `data/oof/ranker_{xgb,cat}_optuna_best*.json` / `data/optuna/ranker_{xgb,cat}_optuna.sqlite3`

### Phase 5: Wide同時確率（PL+MC）→ EV → 資金配分 → バックテスト

**成果物**
- `scripts_v2/backtest_wide_v2.py`
- `data/backtest_result.json`（UI互換）
- `data/backtest_result_meta.json`

**入力**
- `data/oof/top3_oof.parquet`
- DB: `core.o3_wide`（確定オッズ）, `core.payout`（払戻）, `core.result`

**ロジック（PoCベースライン）**
- PL+MCで top3 を非復元抽出し、`p_wide(i,j)` を推定
- オッズは `min_odds_x10/10`（下限）を使用
- EV: `p_wide * odds - 1` を基準にフィルタ
- ベット額: fractional Kelly + 1レース上限 + 100円単位
- 決済: `core.payout(bet_type=5, selection=kumiban)` を用いる

**UI**
- `app/routers/ui.py` は `data/backtest_result.json` を読む前提で、summary/monthly/bets を返す
- `frontend` は既存UIを流用しつつ bets スキーマをワイド用に更新

## 8. リスクと対策（要点）

- DB権限不足（作成/DDL不可）
  - 対策: `sudo -u postgres` でDB作成 → アプリは一般ユーザーで接続運用
- データリーク
  - 対策: as-of検証テストを必須化（Phase 2）
- 品質（stub race / 欠損オッズ / 取消）
  - 対策: 取り込みは保持、学習対象からの除外/補完ルールを Phase 2 で固定化（TODOにメモ済み）
