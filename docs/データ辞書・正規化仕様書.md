* 対象プロジェクト: `keiba`
* 対象:
  * 入力（JV-Data）: `rec_id`（例: `RA`, `SE`, `HR`, `O1` ...）
  * 変換（パース）: `app/infrastructure/parsers.py`
  * 正規化（DB投入）: `scripts/load_to_db.py`
  * DBスキーマ: `migrations/0001_init_db.sql`, `migrations/0002_init_mart.sql`（+ `migrations/*.sql`）
* 最終更新: 2026-02-15

## 1. 目的 / スコープ

### 目的

* JV-Data の **固定長レコード（rec_id）**を、DBの **Raw/Core/Mart** にどう正規化するかを定義する
* 「どのレコードが、どのテーブルの、どのカラムに入るか」を**実装準拠**で追えるようにする

### スコープ（本書で扱う）

* `raw`/`core`/（一部 `mart`）への **レコード → カラム対応表**
* 主要な変換ロジック（キー生成、JOIN/解決、欠損/例外の扱い）

### 非スコープ（参照のみ）

* 取得仕様（JVOpen/JVRTOpenの引数・実行例）: `docs/データ取得仕様書.md`
* 運用の時点固定（T-5採用ルール）: `docs/運用仕様書.md`
* 特徴量仕様・検証設計: `docs/特徴量仕様.md`, `docs/検証設計仕様書.md`

## 2. レイヤとキーの共通ルール

### 2.1 レイヤ

* **Raw（raw）**: 取得したレコードを再パース可能な形で保持（監査・再処理）
* **Core（core）**: 解析・特徴量生成のための正規化（主キー/外部キーを整える）
* **Mart（mart）**: 学習/推論に直接使う特徴量テーブル（本書では CK と T-5スナップを例示）

### 2.2 race_id（共通キー）

実装上の `race_id` は以下で一意生成する（`app/infrastructure/parsers.py` の各パーサで同様の復元を行う）。

```text
date_int = YYYY*10000 + MM*100 + DD
race_id  = date_int*10000 + track_code*100 + race_no
```

例: 2026/02/03 東京(05) 5R → `202602030505`

### 2.3 レコード→DB投入の基本方針

* 原則として **rec_id 単位**で「正規化先テーブル」を決める（例: `RA → core.race`）
* 参照整合性のため、`scripts/load_to_db.py` は
  * **Pass1 で RA を先に投入**（core.race を先に満たす）
  * Pass2 で SE/HR/O1/… を投入
* 外部キー欠損（マスタ欠落等）には「NULL許容 or stub登録」で落ちないようにする（詳細は各レコード節）

## 4. Raw層（再処理・監査）

### 4.1 raw.jv_raw（汎用）

* 投入: `scripts/load_to_db.py`（Pass1でバッチINSERT）

| raw.jv_raw カラム | 由来 | 備考 |
|---|---|---|
| `dataspec` | 入力ファイル名先頭（例: `RACE_...`） | `process_file()` で推定 |

## 5.2 SE（Runner/Result）→ core.runner / core.result

### 対象

* 入力 rec_id: `SE`
* パーサ: `RunnerRecord.parse(payload)`
* ローダ: `upsert_runner(db, runner, master_jockeys, master_trainers)`
* 正規化先: `core.runner`（出走表） / `core.result`（結果）

### 重要: 参照整合性（stub登録・FK回避）

ローダは落ちないことを優先し、以下を行う:

* `core.horse` に horse_id が存在しない場合、`SE` の馬名で **stub登録**する（`ON CONFLICT DO NOTHING`）

* `jockey_id` / `trainer_id` は
  * `data_kubun in ("A","B2")`（地方/海外）なら **NULL** に落とす

## 5.4 O1（Final Odds）→ core.odds_final

### 対象

* 入力 rec_id: `O1`（主に `RACE` dataspec 由来の確定/最終）
* パーサ: `OddsRecord.parse(payload)`（複数行）
* ローダ: `upsert_odds(db, odds)`
* 正規化先: `core.odds_final`

### 重要: horse_id 解決（JOIN）

`O1` レコード自体は horse_id を持たないため、ローダは

* `core.runner` を `race_id + horse_no` で引いて horse_id を解決し、
* `core.odds_final (race_id, horse_id)` にUPSERTする

→ `core.runner` が未投入/欠損だと、当該馬のオッズは `core.odds_final` に入らない。

### マッピング

|---|---|---|
| `race_id` | `race_id` | |
| `horse_id` | `core.runner` から解決 | `horse_no` でJOIN |
| `odds_win` | `odds_1`（bet_type=1） | |
| `pop_win` | `popularity`（bet_type=1） | |
| `odds_place_low` | `odds_1`（bet_type=2） | |
| `odds_place_high` | `odds_2`（bet_type=2） | |
